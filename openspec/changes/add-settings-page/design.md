## Context

当前ESP32风扇控制器的所有配置参数（风扇曲线、温度阈值、INA226参数等）均为`app_config.h`中的`static constexpr`编译期常量。用户无法在运行时调整参数，需要重新编译固件才能修改。

项目现状：
- **UI框架**：LVGL 9.x，由LVGL Editor生成代码（*_gen.c/h）
- **屏幕**：240x280 ST7789 TFT（非触摸）
- **输入**：3个物理按键（K1=上，K2=确认，K3=下），支持短按/长按
- **存储**：ESP32-S3有NVS分区但未使用
- **数据流**：app_state（std::atomic）→ data_bridge（200ms刷新）→ lv_subject → UI控件绑定

## Goals / Non-Goals

**Goals:**
- 提供5个分类设置页面（风扇、温度、显示、功率、传感器）
- 通过3按键实现完整导航和数值编辑
- 使用NVS持久化保存用户设置
- 保持现有代码架构（ui_bridge模式）
- 最小化对现有模块的改动

**Non-Goals:**
- 不实现触摸交互（硬件不支持）
- 不实现WiFi/蓝牙远程配置
- 不实现配置导入/导出
- 不修改现有硬件引脚定义

## Decisions

### 1. 配置管理架构：独立config_manager模块

**决策**：新增`src/app/config_manager.{h,cpp}`，集中管理所有运行时配置。

**理由**：
- 单一职责：配置管理与业务逻辑分离
- 线程安全：使用std::atomic保护多任务访问
- 易于扩展：新增配置项只需添加getter/setter

**替代方案**：
- 直接修改app_config.h为非const → 违反编译期配置原则
- 分散在各模块管理 → 配置分散，难以维护

### 2. 持久化方案：ESP32 Preferences

**决策**：使用Arduino的Preferences库（NVS封装）。

**理由**：
- ESP32原生支持，无需额外依赖
- 简单API：put/get系列函数
- 自动处理Flash写入磨损
- 分区表已有nvs分区

**替代方案**：
- SPIFFS/LittleFS → 需要额外分区，过于复杂
- 自定义NVS → API复杂，无明显优势

### 3. UI实现：LVGL Editor XML + 手写交互逻辑

**决策**：采用两层架构实现设置页面UI。

**第一层：LVGL Editor XML（静态结构）**
- 文件：`ui/screens/settings.xml`
- 作用：定义页面容器结构（标题栏、内容区域、底部提示）
- 导出：用户使用LVGL Editor导出`settings_gen.c/h`
- 内容区域为空容器，由代码动态填充设置项

**第二层：手写LVGL代码（动态交互）**
- 文件：`src/ui_bridge/settings_ui.cpp`
- 作用：实现动态设置项创建、焦点管理、编辑模式、数值调整
- 依赖：`settings_gen.c`提供的`settings_create()`函数
- 复用：生成代码中的样式定义（style_item, style_item_focused等）

**理由**：
- LVGL Editor适合定义静态布局和样式
- 动态交互（焦点轮转、编辑模式、数值绑定）需要手写代码
- 两层分离：XML负责视觉结构，C++负责行为逻辑
- 符合项目现有模式：home.xml定义布局，data_bridge.cpp绑定数据

**替代方案**：
- 完全手写LVGL代码 → 样式管理复杂，不便于可视化设计
- 完全使用LVGL Editor → 无法实现动态交互逻辑

### 4. 页面导航：焦点轮转 + 页面切换

**决策**：
- K1/K3：在设置项间移动焦点
- K2短按：进入编辑模式 / 确认修改
- K2长按：返回上一级（设置主页→主界面，子页面→设置主页）
- 页码显示：页面标题旁显示"1/5"

**理由**：
- 符合"上中下"按键布局的人体工学
- 长按返回是常见嵌入式设备交互模式
- 页码显示直观

**替代方案**：
- 自动翻页 → 用户可能迷失位置
- 组合键 → 记忆负担大

### 5. 数值编辑：闪烁 + 增减

**决策**：
- K2进入编辑模式：当前数值闪烁，显示上下箭头图标
- K1/K3：增加/减少数值
- K2确认：保存修改，退出编辑模式
- 步长：温度0.5°C，百分比1%，功率预设选择

**理由**：
- 闪烁是经典的编辑模式视觉提示
- 步进式调整适合物理按键

**替代方案**：
- 位选编辑 → 3按键操作过于复杂
- 即时调整 → 容易误操作

### 6. 配置项分页

**决策**：
- **第1页：风扇设置**
  - 低温阈值（°C）
  - 中温阈值（°C）
  - 高温阈值（°C）
  - 强制满速阈值（°C）
  - 最低转速（%）
  - 中间转速（%）

- **第2页：温度保护**
  - 过温警告阈值（°C）
  - 过温关机阈值（°C）
  - 滞回带宽（°C）

- **第3页：显示设置**
  - 亮度（0-100%）

- **第4页：功率配置**
  - 设计功率（350/450/550/750W选择）

- **第5页：传感器校准**
  - NTC温度偏移（°C）

**理由**：
- 功能相关项分组
- 5页适合3按键导航
- 每页3-6项，不过多

### 7. 数据流设计

**决策**：
```
用户按键 → input_bridge → settings_ui（焦点/编辑）
                              ↓
                         config_manager::set_xxx()
                              ↓
                         Preferences::put_xxx()（持久化）
                              ↓
fan_curve / fault_guard ← config_manager::get_xxx()
```

**理由**：
- 复用现有input_bridge模式
- config_manager作为单一数据源
- 控制模块读取运行时配置

## Risks / Trade-offs

**风险1：NVS写入寿命**
- **风险**：频繁修改配置可能耗尽NVS写入寿命
- **缓解**：仅在用户确认修改时写入，不自动保存临时调整

**风险2：线程安全**
- **风险**：多任务同时读写配置可能导致数据竞争
- **缓解**：config_manager使用std::atomic或mutex保护

**风险3：UI响应性**
- **风险**：设置页面复杂的焦点管理可能影响LVGL性能
- **缓解**：使用lv_async_call确保UI操作在LVGL线程执行

**风险4：配置项过多**
- **风险**：未来新增配置项可能导致页面臃肿
- **缓解**：预留扩展点，可添加更多分类页面

## Migration Plan

1. **阶段1：config_manager模块**
   - 新增config_manager.{h,cpp}
   - 实现默认值加载
   - 实现NVS读写

2. **阶段2：修改控制模块**
   - fan_curve.cpp读取config_manager
   - fault_guard.cpp读取config_manager
   - data_bridge.cpp使用配置的功率值

3. **阶段3：设置UI**
   - 实现settings_ui.cpp
   - 集成到screen_manager
   - 连接input_bridge

4. **阶段4：测试**
   - 单元测试config_manager
   - 集成测试UI交互
   - 验证NVS持久化
