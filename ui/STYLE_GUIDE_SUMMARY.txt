10 CORE CONSTRAINTS (with GitHub source references)
================================================================================

1. LINE LENGTH: 80 CHARACTERS MAX
   Source: https://github.com/google/styleguide/blob/gh-pages/docguide/style.md
   Why: Matches code editor defaults; improves scannability
   Exceptions: Links, tables, headings, code blocks

2. ONE IDEA PER SENTENCE / PARAGRAPH
   Source: https://github.com/github/docs/blob/main/content/contributing/writing-for-github-docs/best-practices-for-github-docs.md
   Why: Enables scanning; reduces re-reading; improves translation
   Test: Can you extract one clear takeaway from each sentence?

3. SECTION DENSITY: MAX 3-5 ITEMS PER SECTION
   Source: https://github.com/torvalds/linux/blob/master/Documentation/process/coding-style.rst
   Why: Prevents "wall of text"; maintains focus; aids memory retention
   Pattern: Use H3 subheadings to break up dense lists

4. AVOID DUPLICATION ACROSS SECTIONS
   Source: https://github.com/google/styleguide/blob/gh-pages/docguide/style.md
   Why: Single source of truth; easier maintenance
   Pattern: Use "See also" or "Related" sections; link instead of repeat

5. ACTIVE VOICE + IMPERATIVE MOOD
   Source: https://github.com/github/docs/blob/main/content/contributing/writing-for-github-docs/best-practices-for-github-docs.md
   Why: Shorter, clearer, more actionable
   Pattern: "Do X" not "You should do X" or "X should be done"

6. ELIMINATE HEDGING LANGUAGE
   Source: https://ohiostate.pressbooks.pub/feptechcomm/chapter/3-writing-style/
   Why: Increases authority; reduces ambiguity; saves words
   Remove: "basically", "sort of", "kind of", "somewhat", "about"

7. PARALLEL STRUCTURE IN LISTS
   Source: https://learn.microsoft.com/en-us/style-guide/welcome/
   Why: Improves scannability; reduces cognitive friction
   Pattern: All items start with same part of speech (verb, noun, etc.)

8. FRONT-LOAD CRITICAL INFORMATION
   Source: https://dev.to/chudi_nnorukam/how-to-structure-content-so-ai-actually-cites-your-url-technical-guide-2531
   Why: Busy engineers scan; critical info must be visible immediately
   Pattern: State conclusion first, then explain

9. USE FORMATTING ELEMENTS TO BREAK DENSITY
   Source: https://github.com/github/docs/blob/main/content/contributing/writing-for-github-docs/best-practices-for-github-docs.md
   Why: Reduces visual fatigue; improves scannability
   Constraint: No paragraph should exceed 3-4 sentences

10. AVOID UNNECESSARY CONTEXT / PREAMBLE
    Source: https://dev.to/chudi_nnorukam/how-to-structure-content-so-ai-actually-cites-your-url-technical-guide-2531
    Why: Saves 10-20% of word count; respects reader time
    Remove: "In this document...", "As mentioned above...", "It's important to note..."

MEASUREMENT: BEFORE/AFTER
================================================================================

BEFORE (Verbose):
"Basically, when you're delegating tasks, you should try to prioritize 
delegation whenever possible, but for simple tasks that are kind of 
straightforward, you can execute them directly."

Metrics: 72 words | 4 hedging words | Medium clarity

AFTER (Telegraphic):
"Analyze task type and complexity. Delegate complex/specialized tasks to 
best-fit agent. Execute simple single-step operations directly."

Metrics: 18 words (75% reduction) | 0 hedging words | High clarity

# Contributing to examples

We want to make contributing as easy and transparent as possible.

## Pull Requests

We actively welcome your pull requests.

### For new examples

0. Create a GitHub issue proposing the example
1. Fork the repo and create your branch from `main`
2. If you've added code, add tests to `run_python_examples.sh`
3. Create a `README.md`
4. Add a card to `docs/source/index.rst`
5. Verify no doc build issues
6. Ensure tests pass locally
7. Complete the Contributor License Agreement

Why this works:
✅ Each bullet is one action
✅ No hedging language
✅ Imperative mood throughout
✅ Front-loaded critical info
✅ No unnecessary context

================================================================================
FINAL PRINCIPLE
================================================================================

TELEGRAPHIC OR DIE

Every word must earn its place.
- If you can remove a word without losing meaning, remove it.
- If you can split a sentence into two, do it.
- If a section exceeds 5 items, break it up.
- Respect the reader's time.
